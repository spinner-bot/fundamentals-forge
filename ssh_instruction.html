1. 🔐 SSH 核心原理

SSH（Secure Shell）不是"一个加密的命令行工具"，而是一套完整的网络协议。
它工作在 TCP/IP 协议栈的应用层，默认端口 22，为不安全的网络提供安全的远程登录、命令执行和文件传输能力。

1.2 加密基础：两种加密的巧妙配合

SSH 的安全性建立在对称加密和非对称加密的混合使用上 — 各取所长。
特性对称加密非对称加密
密钥数量1 把密钥（加解密用同一把）2 把密钥（公钥加密，私钥解密）
速度⚡ 非常快🐢 慢 100~1000 倍
密钥分发❌ 难题：如何安全地把密钥给对方？✅ 简单：公钥可以随便公开
SSH 中的角色💪 加密真正的会话数据（传输效率高）🔑 验证身份 + 安全协商对称密钥
典型算法AES-256-GCM, ChaCha20-Poly1305RSA, Ed25519, ECDSA
💡
SSH 的精妙设计：用非对称加密做"钥匙交换"（安全但慢），商定一把临时的对称密钥后，后续所有数据都用对称加密传输（快速而安全）。这就是混合加密的思想。

1.3 SSH 连接的完整过程（分三阶段）

阶段一：TCP 连接 + 协议版本协商

Client ──TCP SYN──→ Server (port 22)
Client ←─TCP SYN/ACK── Server
Client ──TCP ACK──→ Server

# 双方交换协议版本字符串
Client → "SSH-2.0-OpenSSH_9.6"
Server → "SSH-2.0-OpenSSH_9.3"

阶段二：密钥交换（最关键的安全基础）

sequenceDiagram participant C as "💻 客户端" participant S as "🖥️ 服务器" Note over C,S: 密钥交换（DH 算法） C->>S: 发送支持的加密算法列表 S->>C: 选择算法 + 发送服务器公钥(Host Key) C->>C: 验证 Host Key 指纹 C->>S: DH 密钥交换参数 S->>C: DH 密钥交换响应 Note over C,S: 🔑 双方独立计算出相同的 Session Key Note over C,S: ✅ 后续全部用 Session Key 对称加密

DH（Diffie-Hellman）密钥交换的核心魔法：双方在不安全的信道上各自做数学运算，最终得出相同的共享密钥—而窃听者即使看到所有通信内容也无法计算出这个密钥。

阶段三：用户认证

认证方式原理安全性推荐度
密码认证客户端把密码加密后发给服务器，服务器验证⚠️ 中等：密码可被暴力破解❌ 不推荐
公钥认证（⭐）服务器用你的公钥加密一个随机数，只有你能用私钥解密并返回✅ 高：无法暴力破解私钥⭐ 强烈推荐
Keyboard-Interactive服务器提问，客户端回答（常用于 2FA）✅ 高：可结合 TOTP✅ 推荐
GSSAPI基于 Kerberos 的企业单点登录✅ 高企业环境

公钥认证详细过程

sequenceDiagram participant C as "客户端" participant S as "服务器" C->>S: 1. 发送认证请求（含公钥指纹） S->>S: 2. 查 ~/.ssh/authorized_keys 匹配公钥 S->>C: 3. "请证明你有对应的私钥"（发送随机挑战） C->>C: 4. 用本地私钥对挑战做数字签名 C->>S: 5. 返回签名结果 S->>S: 6. 用公钥验证签名 S->>C: 7. ✅ 认证通过 / ❌ 拒绝
⚠️
Host Key 验证的意义：第一次连接时 SSH 会让你确认服务器的指纹 — 这不是走形式！它是防止中间人攻击（MITM）的唯一防线。
如果你点了"yes"但连到了黑客的假服务器，后面输入的所有内容（包括密码和密钥）都可能被窃取。
正确做法：第一次连接前，通过安全渠道（当面/电话/内部 wiki）获取服务器的指纹并手动比对。

2. 🔑 密钥生成与管理

🎯 ssh-keygen 密钥生成 基础

💡 一句话：生成公私钥对 — 私钥自己保管（等同于密码），公钥放到服务器上。

密钥类型对比

类型安全性速度密钥大小推荐?
ed25519🔒🔒🔒 极高⚡ 快~68 字符（极小）⭐ 强烈推荐
ecdsa🔒🔒 中等⚡ 快较小✅ 可用
rsa 4096🔒🔒🔒 高🐢 较慢~740 字符✅ 兼容性最好
rsa 2048🔒🔒 中等🐢 较慢~380 字符⚠️ 不推荐（已不够安全）
dsa🔒 低——❌ 已废弃

生成命令（推荐 Ed25519）

# 最推荐：Ed25519（安全 + 快速 + 密钥小）
ssh-keygen -t ed25519 -C "your_email@example.com"

# 兼容性最好：RSA 4096（旧服务器可能不支持 Ed25519）
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"

# 指定输出文件（多密钥管理）
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_lab -C "lab_server_key"

# 参数说明：
# -t : 密钥类型
# -b : 密钥位数（RSA用）
# -C : 注释（方便识别，一般填邮箱）
# -f : 输出文件路径
🎯 公钥部署到服务器 基础

💡 一句话：把你的公钥内容追加到服务器上 ~/.ssh/authorized_keys 文件中。

# 方法一：一键复制（推荐，自动处理权限）
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server

# 方法二：手动复制（当没有 ssh-copy-id 时）
cat ~/.ssh/id_ed25519.pub | ssh user@server "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"

# 方法三：完全手动
# 1. 查看本地公钥
cat ~/.ssh/id_ed25519.pub
# 2. SSH 密码登录服务器后，把公钥内容追加到 authorized_keys
echo "ssh-ed25519 AAAAC3NzaC1lZ..." >> ~/.ssh/authorized_keys

# ⚠️ 重要：确保权限正确！
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
🔴
权限是 SSH 公钥认证的常见陷阱：
• ~/.ssh/ 必须是 700（只有你能读写）
• ~/.ssh/authorized_keys 必须是 600（只有你能读写）
• 权限不对 → SSH 会静默忽略你的公钥，回退到密码认证（或不让你登录）
• 排查时看 ssh -vvv 的输出中是否有权限警告
🎯 ssh-agent：密钥管家 进阶

💡 一句话：ssh-agent 在后台帮你记住私钥的密码（passphrase），这样每天只需要输入一次密码即可反复使用密钥。

# 启动 agent（现代系统通常自动启动）
eval "$(ssh-agent -s)"

# 添加密钥到 agent（输入一次 passphrase 后由 agent 托管）
ssh-add ~/.ssh/id_ed25519

# 列出 agent 中已加载的密钥
ssh-add -l

# 删除所有已加载的密钥
ssh-add -D

# 锁定/解锁 agent（让已加载的密钥暂时不可用）
ssh-add -x        # 锁定
ssh-add -X        # 解锁

# macOS：自动记住 passphrase 到钥匙串
ssh-add --apple-use-keychain ~/.ssh/id_ed25519
💡
passphrase 是什么？生成密钥时可以给它加一层密码保护。即使私钥文件泄露，没有 passphrase 也无法使用。ssh-agent 让你不用每次都输入它。

3. ⚙️ 客户端配置（~/.ssh/config）

~/.ssh/config 是 SSH 最被低估但最强大的功能之一。它让你用简短的别名替代一长串命令，同时透明地处理跳板机、端口转发、密钥选择等。
🎯 配置文件语法与基础项 核心

💡 一句话：Host 定义一组配置，缩进（可选）表示从属关系，匹配到的第一个 Host 生效。

# ~/.ssh/config 完整示例
# 全局默认值（对所有 Host 生效）
Host *
    ServerAliveInterval 60          # 每 60s 发心跳包防断连
    ServerAliveCountMax 5           # 连续 5 次无响应才判定断开
    TCPKeepAlive yes
    Compression yes                 # 开启压缩（慢网络有用）
    IdentitiesOnly yes              # 只用显式指定的密钥

# 实验室跳板机
Host bastion
    HostName 202.120.xx.xx          # 跳板机公网 IP 或域名
    Port 22
    User zhang-san                   # 你的用户名
    IdentityFile ~/.ssh/id_ed25519
    ServerAliveInterval 30

# 内网 GPU 服务器（通过跳板机连接）
Host gpu01
    HostName 192.168.1.101           # 内网 IP
    User zhang-san
    IdentityFile ~/.ssh/id_ed25519
    ProxyJump bastion                # ⭐ 通过跳板机
    # 等价旧写法: ProxyCommand ssh -W %h:%p bastion

Host 匹配规则（重要！）

模式匹配示例
Host myserver精确匹配ssh myserver
Host gpu*通配符匹配gpu01, gpu02, gpu-train
Host *匹配所有（放最后做默认值）全局设置
Host !pattern排除匹配Host * !bastion

🎯 免密登录链 进阶

💡 一句话：跳板机+内网服务器的完整免密链，需要把公钥分别部署到每一跳。

# 步骤 1: 生成密钥（只需一次）
ssh-keygen -t ed25519 -C "lab-key"

# 步骤 2: 把公钥放到跳板机
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@bastion

# 步骤 3: 把公钥也放到内网服务器
# 方法 A — 从本地直接放（如果网络可达）
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@gpu01

# 方法 B — 从跳板机转发（如果内网不通本地）
ssh bastion "cat >> ~/.ssh/authorized_keys" < ~/.ssh/id_ed25519.pub
# 然后从跳板机 ssh-copy-id 到内网
ssh bastion
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@gpu01

# 验证免密链
ssh gpu01   # 应直接登录（本机→跳板机→内网服务器，全程免密）

4. 📁 文件传输

🎯 scp（安全复制） 基础

💡 一句话：基于 SSH 的安全文件复制命令，语法类似 cp，支持本地↔远程双向。

# 基本语法
scp [选项] 源 目标

# 上传：本地 → 远程
scp data.csv user@server:/home/user/data/
scp -r model/ user@server:/home/user/          # -r 递归复制目录

# 下载：远程 → 本地
scp user@server:/logs/app.log ./
scp -r user@server:/home/user/results/ ./      # 下载整个目录

# 常用选项
scp -P 2222 file user@server:/path/            # 指定端口（大写 P！）
scp -C large_file user@server:/path/            # 压缩传输
scp -v file user@server:/path/                  # 显示详细信息（调试用）
scp -p file user@server:/path/                  # 保留文件时间戳和权限

通过跳板机传输文件

# 方法 1：如果你配置了 ~/.ssh/config 中的 ProxyJump
scp data.csv gpu01:/home/user/data/    # 直接用别名！与普通 scp 完全一样

# 方法 2：显式指定跳板机
scp -o ProxyJump=bastion data.csv gpu01:/home/user/

# 方法 3：通过跳板机中转（笨办法，但能理解原理）
scp data.csv bastion:/tmp/             # 先传到跳板机
ssh bastion "scp /tmp/data.csv gpu01:/home/user/"  # 再从跳板机传到内网
🎯 rsync（增量同步） 进阶

💡 一句话：比 scp 更智能的同步工具 — 只传差异部分，支持断点续传，适合大数据集和代码同步。

# 基本用法（-a 归档模式 = 保留权限+时间戳+递归, -v 详细, -z 压缩）
rsync -avz src_dir/ user@server:dst_dir/

# ⚠️ 注意：src/ 和 src 的区别！
rsync -avz data/ user@server:data/     # 同步 data/ 目录里的内容 到远程 data/
rsync -avz data user@server:data/      # 同步 data 目录本身 到远程

# 常用组合
rsync -avz --progress data/ user@server:data/   # 显示进度
rsync -avz --delete src/ dst/                    # ⚠️ 镜像同步（目标多余文件会被删）
rsync -avz --exclude='*.pyc' --exclude='__pycache__/' src/ dst/  # 排除文件
rsync -avz --max-size=100m src/ dst/             # 只传 ≤100MB 的文件
rsync -avz --dry-run src/ dst/                   # 试运行：只显示会做什么，不实际执行
rsync -avz -e "ssh -p 2222" src/ user@host:dst/ # 指定 SSH 选项

# 通过跳板机同步
rsync -avz -e "ssh -J bastion" src/ gpu01:dst/  # -J 指定跳板机
# 或者配置了 ProxyJump 后直接用别名
rsync -avz src/ gpu01:dst/

scp vs rsync 选哪个？

场景推荐原因
传几个小文件scp简单直接
传大文件 / 数据集rsync断点续传，不怕中断
同步项目代码rsync只传变更部分，快得多
定期备份目录rsync增量+去重，效率最高
简单的一次性复制scp少打几个字母
🎯 sftp（交互式文件管理） 实用

💡 一句话：基于 SSH 的交互式文件管理器，类似 FTP，支持 ls/cd/get/put 等命令。许多 GUI 文件管理器也基于 SFTP。

# 连接到服务器
sftp user@server

# sftp 内的常用命令
sftp> ls                     # 列出远程目录
sftp> cd /remote/path        # 切换远程目录
sftp> lcd /local/path        # 切换本地目录
sftp> get file.txt           # 下载（远程→本地）
sftp> get -r dir/            # 递归下载目录
sftp> put file.txt           # 上传（本地→远程）
sftp> put -r dir/            # 递归上传
sftp> rm file.txt            # 删除远程文件
sftp> pwd                    # 显示远程当前目录
sftp> lpwd                   # 显示本地当前目录
sftp> !command               # 执行本地命令（如 !ls）

# 通过跳板机也可以
sftp -J bastion gpu01

5. 🏗️ 跳板机原理

跳板机（Jump Host / Bastion Host）是内外网之间的安全桥梁。它不是"多了一步麻烦的障碍"——而是在"公网可达"和"内网安全"之间做的一个必要权衡。

5.2 跳板机的安全意义

如果没有跳板机有跳板机
❌ 每台服务器都暴露在公网 → 每个都是攻击目标✅ 只有一台暴露在公网，其余隐藏在内网
❌ SSH 端口被全网扫描，暴力破解 24/7 不间断✅ 内网服务器完全不受公网扫描影响
❌ 任何一台被攻破 → 内网全部沦陷✅ 跳板机成为控制点，可以集中审计和防护
❌ 修改防火墙规则要逐台操作✅ 只需管理跳板机的访问控制
❌ 难以审计谁在什么时间登录了哪台机器✅ 所有 SSH 会话必经跳板机 → 集中审计

5.3 跳板机的工作模式

sequenceDiagram participant C as "💻 你的电脑" participant J as "🛡️ 跳板机" participant T as "🖥️ 内网服务器" Note over C,T: 方式一：Jump（推荐 — 透明隧道） C->>J: 1. SSH 连接跳板机（认证） C->>J: 2. 请求：请帮我转发到 gpu01:22 J->>T: 3. 跳板机向内网 gpu01 发起 TCP 连接 T-->>J: 4. gpu01 的 SSH 服务响应 Note over C,T: 5. SSH 连接在 C↔T 之间端到端加密建立！ Note over C,T: ⭐ 跳板机只做转发，看不到加密内容
🔑
关键认知：跳板机看不到你的数据
使用 ProxyJump（-J）时，SSH 连接在你和最终服务器之间端到端加密。跳板机只是在 TCP 层面做数据转发，无法看到 SSH 会话的内容。这与先 SSH 到跳板机再手动 SSH 到内网（跳板机可以看到你的操作）有本质区别。

8. 🌐 端口转发与隧道

SSH 不仅能转发命令行 — 它可以转发任何 TCP 流量。这是做内网穿透、远程调试、安全代理的利器。

8.1 三种转发模式对比

🎯 本地转发 (-L)：把远程端口"搬"到本地 核心

💡 一句话：访问你本机的 localhost:某个端口，就等于访问远程服务器的某个端口。最常见的用法：远程 Jupyter / Web 服务 / 数据库。

格式

ssh -L [本地地址:]本地端口:目标地址:目标端口 [跳板机] 最终服务器

实战场景

# 场景 1：访问内网服务器的 Jupyter（经典！）
ssh -N -L 8888:localhost:8888 gpu01
# 解释：本机 8888 → gpu01 的 localhost:8888

# 场景 2：访问内网另一台机器（不是 gpu01 本身）的 Web 服务
ssh -N -L 8080:192.168.1.50:80 gpu01
# 解释：本机 8080 → 通过 gpu01 → 192.168.1.50:80

# 场景 3：通过跳板机访问内网的 MySQL 数据库
ssh -N -L 3306:192.168.1.100:3306 bastion
# 然后在本地用 mysql -h 127.0.0.1 -P 3306 连接

# 场景 4：同时转发多个端口
ssh -N -L 8888:localhost:8888 -L 6006:localhost:6006 gpu01

# -N 参数：不执行远程命令，只做端口转发（很重要）
🎯 远程转发 (-R)：把你本地服务"暴露"到远程 进阶

💡 一句话：让远程服务器上的某个端口接收的请求，通过 SSH 隧道转发到你本地。常用于：让远程同事访问你本机的开发环境；或让内网服务"回连"到你的电脑。

# 场景：你本机跑了一个 Web 服务在 localhost:3000
# 想让远程服务器 gpu01 的 9090 端口也能访问它

ssh -R 9090:localhost:3000 gpu01
# 之后在 gpu01 上访问 localhost:9090 = 你本机的 localhost:3000

# ⚠️ 默认只绑定到 localhost，要让其他机器也能访问：
# 需要在服务器端 /etc/ssh/sshd_config 设置：
# GatewayPorts yes
# 然后：
ssh -R 0.0.0.0:9090:localhost:3000 gpu01
# 这样内网其他机器也能通过 gpu01:9090 访问你本机的服务
🎯 动态转发 (-D)：SOCKS 代理 进阶

💡 一句话：在你本机开一个 SOCKS5 代理端口，把浏览器/APP 的流量通过跳板机转发。相当于把你的电脑"变成"在内网中。

# 在本机开启 SOCKS 代理
ssh -D 1080 -N bastion

# 然后配置你的浏览器/应用使用 SOCKS5 代理：
#   地址：localhost
#   端口：1080
#   类型：SOCKS5

# 使用场景举例：
# • 访问只能在学校内网打开的学术资源/数据库
# • 通过实验室网络下载论文（IP 认证的期刊）
# • 调试只能内网访问的 Web 应用

# ⭐ Chrome 快捷方式（只代理当前窗口）
# chrome --proxy-server="socks5://localhost:1080" --user-data-dir=/tmp/chrome-ssh

端口转发对照表

类型参数数据流最常用于
本地转发-L 本地:目标:端口本地监听 → 远程目标⭐ Jupyter、TensorBoard、数据库
远程转发-R 远程:目标:端口远程监听 → 本地目标内网穿透、远程调试本地服务
动态转发-D 本地端口SOCKS → 远程任意目标代理上网、访问内网资源

10. 🛡️ 安全加固

SSH 暴露在公网 = 全天候被全球攻击者扫描和暴力破解。以下措施不是可选项，是必选项。
🎯 服务端安全配置（/etc/ssh/sshd_config） 重要

💡 一句话：修改 sshd 配置后必须 sudo systemctl reload sshd。改之前保持另一个 SSH 窗口打开，防止把自己锁在外面！

# === 最核心的安全设置 ===
# 禁止 root 直接 SSH 登录（用普通用户登录后 sudo）
PermitRootLogin no

# 禁用密码认证（只允许密钥登录！）
PasswordAuthentication no
ChallengeResponseAuthentication no

# 只允许特定用户/组登录
AllowUsers zhang-san lisi
AllowGroups ssh-users

# 限制认证尝试次数
MaxAuthTries 3

# 空闲超时自动断开（秒）
ClientAliveInterval 300
ClientAliveCountMax 2

# === 其他强化 ===
# 限制同时连接数
MaxSessions 5
MaxStartups 10:30:100

# 关闭不用的功能
X11Forwarding no                     # 不需要 X11 就关
AllowTcpForwarding yes               # 需要端口转发则保留

# 修改后重载
sudo systemctl reload sshd
🎯 fail2ban：自动封禁暴力破解 IP 进阶

💡 一句话：fail2ban 监控 SSH 日志，发现某个 IP 短时间内多次认证失败 → 自动在防火墙封禁该 IP。从被动防护变为主动防御。

# 安装
sudo apt install fail2ban

# 创建本地配置（不要直接改默认配置）
sudo cp /etc/fail2ban/jail.conf /etc/fail2ban/jail.local

# 编辑 /etc/fail2ban/jail.local，确保以下配置生效：
[sshd]
enabled = true
port = ssh
filter = sshd
logpath = /var/log/auth.log
maxretry = 5                 # 5 次失败后封禁
bantime = 3600               # 封禁 1 小时（单位秒）
findtime = 600               # 10 分钟内
ignoreip = 127.0.0.1/8 202.120.xx.xx/32  # 白名单

# 启动
sudo systemctl enable fail2ban
sudo systemctl start fail2ban

# 查看被禁 IP
sudo fail2ban-client status sshd
sudo fail2ban-client get sshd banned

# 手动解封某个 IP
sudo fail2ban-client set sshd unbanip 1.2.3.4
🎯 安全审计与日志 进阶
检查项命令说明
查看登录成功记录sudo grep "Accepted" /var/log/auth.log谁、什么时间、从哪个 IP 登录成功
查看登录失败记录sudo grep "Failed" /var/log/auth.log暴力破解痕迹
查看当前登录用户who 或 w谁现在连着
查看历史登录last 或 lastlog所有用户的登录历史
踢掉某个 SSH 会话sudo pkill -u username sshd断开指定用户的所有 SSH
查看失败 IP 排行sudo grep "Failed password" /var/log/auth.log | grep -oP 'from \K\S+' | sort | uniq -c | sort -rn | head找出暴力破解源 IP Top 10
🎯 个人使用安全守则 必读
守则为什么
🔑 私钥设密码（passphrase）即使私钥文件泄露，没有 passphrase 也无法使用
🔒 私钥永远不分享、不上传私钥 = 你的密码，泄露 = 所有部署该公钥的服务器都不安全
📋 不用 ForwardAgent，除非确有必要ForwardAgent 让远程服务器可以使用你本地的 ssh-agent。如果远程服务器被黑，攻击者可以冒充你登录其他机器
🔍 定期检查 authorized_keys确保没有你不再使用的旧公钥残留
🚫 不在公用电脑上存私钥实验室公用电脑上的私钥 = 所有人的私钥
🔄 定期更换密钥建议至少每年更换一次

# 基本调试（输出非常详细，包括每一步的认证过程）
ssh -vvv user@server

# 跳板机调试

# 关键日志行解读：
# debug1: Connecting to 192.168.1.101 [192.168.1.101] port 22.
#   → TCP 连接成功
# debug1: Connection established.
#   → SSH 连接建立
# debug1: Host 'xxx' is known and matches the ED25519 host key.
#   → 主机密钥验证通过
# debug1: Authentications that can continue: publickey,password
#   → 服务器支持的认证方式
# debug1: Trying private key: /home/user/.ssh/id_ed25519

🎯 连接超时排查流程 实用 
# 1. 确认网络可达
ping 202.120.xx.xx                  # 能通 = 网络层 OK

# 2. 确认端口开放
nc -zv 202.120.xx.xx 22             # Connection succeeded! = 端口 OK
# 或
telnet 202.120.xx.xx 22             # 看到 SSH-2.0-OpenSSH... = 端口 OK

# 3. 带详细日志连接

# 4. 如果在学校外，先确保装了 VPN 且能连到学校网络
# 5. 如果是跳板机模式，先测试能不能直接连跳板机
ssh bastion                         # 第一跳能通？
# 6. 跳板机能通，那从跳板机直接试内网
ssh bastion "nc -zv 192.168.1.101 22"  # 跳板机能连内网吗？

# 7. 查看服务器日志
sudo tail -f /var/log/auth.log     # 实时看认证日志
sudo journalctl -u ssh -f           # systemd 的 SSH 日志

B. 📁 典型配置模板集合

B.1 实验室完整 ~/.ssh/config 模板

# ===== 全局默认 =====
Host *
    ServerAliveInterval 60
    ServerAliveCountMax 5
    TCPKeepAlive yes
    IdentitiesOnly yes
    StrictHostKeyChecking accept-new
    ConnectTimeout 15

# ===== 跳板机 =====
Host bastion
    HostName 202.120.xx.xx
    Port 22
    User zhang-san
    IdentityFile ~/.ssh/id_ed25519_lab

# ===== 内网 GPU 服务器（通过跳板机）=====
Host gpu01
    HostName 192.168.1.101
    User zhang-san
    IdentityFile ~/.ssh/id_ed25519_lab
    ProxyJump bastion
    # 自动转发常用端口
    LocalForward 8888 localhost:8888     # Jupyter
    LocalForward 6006 localhost:6006     # TensorBoard
    LocalForward 8265 localhost:8265     # Ray Dashboard

Host gpu02
    HostName 192.168.1.102
    User zhang-san
    IdentityFile ~/.ssh/id_ed25519_lab
    ProxyJump bastion

# ===== 存储服务器（两级跳板）=====
Host storage
    HostName 192.168.2.10
    User zhang-san
    IdentityFile ~/.ssh/id_ed25519_lab
    ProxyJump gpu01      # 本机→bastion→gpu01→storage

# ===== GitHub（使用独立密钥）=====
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_github
    IdentitiesOnly yes
