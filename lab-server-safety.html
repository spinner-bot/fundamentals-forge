1. 🚨 安全红线 13 条

以下任何一条的违反，都可能导致服务器宕机、全组数据丢失、甚至硬件损坏。
每一条红线后面都标注了「违反后果」——这不是吓唬你，这些都是真实发生过的事故。

🔴 红线 #1：严禁 sudo rm -rf / 或任何涉及系统目录的强制删除

禁止示例：

sudo rm -rf /              # ☠️ 删除整个系统 → 服务器立刻变砖
sudo rm -rf /etc/*          # ☠️ 删除所有配置文件 → 系统不可用
sudo rm -rf /boot/*         # ☠️ 删除内核 → 无法启动，需重装系统
sudo rm -rf /home/*         # ☠️ 删除所有用户主目录 → 全组数据灭失

违反后果：服务器不可逆损坏，需请 IT 部门重装系统，全组停工 3-7 天，所有未备份数据永久丢失。

防护措施：

  • 执行 rm 之前，养成先说一遍完整路径的习惯：「我要删除 /data/tmp/下的 old_logs」
  • 绝对禁止：在 sudo 下使用 rm -rf + 通配符（如 rm -rf /tmp/*）
  • 推荐用 trash-cli（回收站工具）：pip install trash-cli; trash-put file

🔴 红线 #2：严禁强制重启服务器

禁止操作：

sudo reboot                 # ☠️ 直接重启 → 所有人的训练进程被强制中断
sudo shutdown -r now        # ☠️ 同上
sudo reboot -f              # ☠️ 强制重启（跳过所有优雅关闭过程，可能损坏文件系统）

违反后果：所有同学正在运行的训练（可能已跑了几周）全部报废，文件系统可能损坏，磁盘需要 fsck 修复。

正确做法：如果服务器确实需要重启（如内核安全更新），必须：
① 提前 24 小时在全组群通知
② 确认所有人的训练已保存 checkpoint
③ 由管理员执行重启操作

🔴 红线 #3：严禁占满磁盘（/ 或 /home 分区）

磁盘占满的危害：

  • 操作系统无法写入日志 → 服务异常
  • SSH 无法登录（sshd 需要写日志）
  • 数据库/调度器停止工作 → 全组无法使用
  • 用户自己的训练 checkpoint 写入失败 → 训练白跑

违反后果：服务器不可用，需管理员介入清理，可能需物理接触服务器才能抢救。

防护：df -h 每周至少看一次；个人数据不要超过配额；不把大规模数据集直接放在 /home 下。

🔴 红线 #4：严禁修改系统级文件（除非你是管理员并清楚后果）

# ☠️ 禁止随意修改的文件/目录
/etc/fstab                 # 改了可能导致无法启动
/etc/ssh/sshd_config       # 改错可能导致所有人（包括你）无法登录
/etc/sudoers               # 改错可能导致无法 sudo
/boot/*                    # 内核文件——碰都不要碰
/usr/bin/*                 # 系统命令——不要替换或删除
/lib/*                     # 系统库——删错会导致系统命令全部不可用

违反后果：轻则部分功能不可用，重则服务器需要重装。

原则：如果不确定一个文件是干什么的——不要改它。先查、先问。

🔴 红线 #5：严禁用 root 跑日常任务

# ☠️ 危险行为
sudo python train.py       # 脚本中的 bug 可能损坏系统
sudo pip install xxx        # 可能覆盖系统的 Python 包
sudo jupyter notebook       # web 服务以 root 运行 → 巨大安全漏洞

# ✅ 正确做法
python train.py             # 用普通用户权限运行
pip install --user xxx      # 安装到用户目录

违反后果：代码中的 bug（如无限写日志）在 root 权限下可能造成不可挽回的系统损坏。root 权限就像拿着一把上了膛的枪——没有必要时，不要拿它。

🔴 红线 #6：严禁占用全部 CPU 核心跑多进程（除非确认无人使用）

# ☠️ 危险行为
python train.py --num_workers=64    # DataLoader 开 64 个进程霸占全部 CPU
# 或
for i in {1..100}; do python task.py & done   # 一次性 fork 100 个后台任务

# ✅ 正确做法
# 先用 htop/top 确认 CPU 空闲核心数
# DataLoader workers 不要超过 CPU 核心数的一半
python train.py --num_workers=8

违反后果：其他同学的 SSH 终端卡顿、训练变慢、甚至整机响应超时被误判为宕机。

🔴 红线 #7：严禁占用全部内存（RAM）

# ☠️ 危险行为
import numpy as np
huge_array = np.zeros((500000, 500000))  # 试图分配 2TB —— OOM Killer 出动

# 内存耗尽 → Linux OOM Killer 随机杀进程
# 可能杀掉别人的训练、SSH 服务、甚至数据库

违反后果：触发 OOM Killer，随机杀掉系统中的进程——可能是你自己的，也可能是别人的关键服务。

🔴 红线 #8：严禁在共享服务器上跑死循环 / fork 炸弹

# ☠️ fork 炸弹（绝对禁止！）
:(){ :|:& };:

# ☠️ 无限制写文件
while true; do echo "log" >> /home/user/huge.log; done   # → 占满磁盘

违反后果：CPU 100% → 整机瘫痪；磁盘占满 → 服务器不可用。

🔴 红线 #9：严禁在未指定 GPU 的情况下跑大模型训练

# ☠️ PyTorch 默认使用 GPU 0！
# 如果你不指定，所有人的训练都挤在 GPU 0 上，而 GPU 1-7 空着

# ✅ 必须显式指定
CUDA_VISIBLE_DEVICES=3 python train.py

违反后果：GPU 0 被所有人挤爆（OOM），而其他 GPU 白白浪费。一台 8 卡服务器被人用成了单卡服务器。

🔴 红线 #10：严禁在 /tmp 下放重要文件

/tmp 在大多数 Linux 系统中重启后会被清空。不要把训练 checkpoint、代码、脚本放在这里。

🔴 红线 #11：严禁长期霸占 GPU 显存不释放

# ☠️ 错误示范
python train.py   # 训练跑完了，但进程没退出 → GPU 显存一直被占用
# 或者：开着 jupyter notebook，kernel 占着 20GB 显存，三天不用

# ✅ 正确做法
# 训练结束 → 立刻退出 Python 进程
# Jupyter 用完 → Kernel → Shutdown
# 定期检查自己的 GPU 进程
gpustat -apu | grep $(whoami)

违反后果：GPU 显存被空进程占着，真正需要训练的人用不了。一周下来相当于浪费了几十小时的 GPU 时间。

🔴 红线 #12：严禁不打招呼就 kill 别人的进程

如果发现某块 GPU 被占着但利用率很低（可能是 bug），先在群里问一声。不要直接 kill。

正确流程：gpustat 查看 → 找到进程主人的用户名 → 在群里 @对方：「同学你的 GPU 利用率显示 0%，确认下是否还在用？」→ 对方回复后再操作。

🔴 红线 #13：严禁在服务器上下载盗版 / 涉密 / 违规内容

实验室服务器的 IP 通常在学校校园网内。下载盗版软件、论文代写工具、翻墙工具、涉密资料等行为可能导致学校网络中心封锁 IP，全组无法访问服务器。

2. 📁 文件与磁盘操作规范

2.1 推荐目录结构

/home/用户名/               # 你的主目录：代码、配置、小型脚本
├── projects/               # 各项目代码
│   ├── project-A/
│   │   ├── code/
│   │   ├── checkpoints/    # ⚠️ 定期清理旧 checkpoint！
│   │   └── logs/
│   └── project-B/
├── .ssh/                   # SSH 密钥（权限必须 700/600！）
├── .bashrc                 # 个人环境配置
└── .conda/                 # conda 环境（也可以放 /data 下）

/data/用户名/               # ⭐ 大数据集、大型模型文件放这里
├── datasets/
├── pretrained_models/
└── large_outputs/

# ⚠️ 不要把 ImageNet/COCO 这样的大数据集直接解压到 /home！
# 100GB 数据集放到 /home 可能占满分区，让全组无法登录

2.2 磁盘监控（每周必做）

# ① 查看各分区剩余空间
df -h
# 重点看 / 和 /home 的 Use% 是否超过 80%

# ② 查看自己占了多大空间
du -sh /home/$(whoami) /data/$(whoami) 2>/dev/null

# ③ 找出自己目录下的 Top 10 大文件
du -ah /home/$(whoami) 2>/dev/null | sort -rh | head -10

# ④ 查看 inode 使用（小文件太多也会耗尽文件系统）
df -i

# ⑤ 找大文件（>1GB）
find /home/$(whoami) -type f -size +1G -exec ls -lh {} \; 2>/dev/null

2.3 清理策略

文件类型保留策略说明
模型 checkpoint只保留 best + 最近 3 个 epoch每个 checkpoint 可能几百 MB~几 GB
训练日志保留 30 天，之后压缩归档gzip *.log 压缩率可达 90%
TensorBoard event 文件任务结束后删除这类文件非常占空间
下载的数据集压缩包解压后删除压缩包.tar.gz 和内容同时存在 = 占双倍空间
Python cache 文件随时可删find . -type d -name __pycache__ -exec rm -rf {} +
conda 缓存每月清理conda clean --all
pip 缓存每月清理pip cache purge
✅ 好习惯：在 checkpoint 保存代码中只保留最优模型 + 最近 3 个： 
# 每个 epoch 保存后，删除旧的 checkpoint
import os, glob
ckpts = sorted(glob.glob('ckpt_epoch_*.pth'))
for old in ckpts[:-3]:   # 只保留最近 3 个
    os.remove(old)

3. 🖥️ GPU 使用规范

3.1 使用前必查：选一块空闲的 GPU

# ⭐ 最推荐：gpustat（信息最全）
# 先安装：pip install gpustat
gpustat -apu
# 输出示例：
# gpu01  Fri Jul 10 15:00:00 2025  550.90.07
# [0] GeForce RTX 4090 | 35°C,  95 % | 22000 / 24564 MB | zhang-san:train.py/12345(21500MB)
# [1] GeForce RTX 4090 | 40°C,   0 % |     0 / 24564 MB | (无进程)
#   → GPU 1 空闲，用 CUDA_VISIBLE_DEVICES=1

# 也可以看详细信息
nvidia-smi

# 查看 GPU 利用率（谁在用但实际没跑任务）
nvidia-smi --query-gpu=index,utilization.gpu,memory.used,name --format=csv

3.2 指定 GPU 的四种方式

# 方式一：命令行指定（⭐ 最推荐，每次运行显式指定）
CUDA_VISIBLE_DEVICES=2 python train.py

# 方式二：在代码开头指定（必须在 import torch 之前）
import os; os.environ['CUDA_VISIBLE_DEVICES'] = '2'
import torch

# 方式三：临时对当前 shell 生效
export CUDA_VISIBLE_DEVICES=0,1    # 使用 GPU 0 和 1

# 方式四：在 .bashrc 中设置默认值（⚠️ 容易忘记改，不推荐）
# echo 'export CUDA_VISIBLE_DEVICES=1' >> ~/.bashrc

3.3 显存管理最佳实践

技巧代码效果
混合精度训练with torch.amp.autocast('cuda'):显存减少 ~40%，速度提升 30-50%
梯度累积loss.backward() 每 N 步才 optimizer.step()小显存也能跑大 batch
手动释放显存del var; torch.cuda.empty_cache()释放不再用的显存
梯度检查点from torch.utils.checkpoint import checkpoint用计算换显存，节省 50%+
batch_size 选择先用小 batch_size 跑通，再逐渐加大避免一开始就 OOM
推理时 no_gradwith torch.no_grad():不存计算图，大幅省显存

3.4 多用户 GPU 使用礼仪

⚠️ GPU 使用黄金法则：
  1. 先查后用：每次跑训练前必须 gpustat 确认 GPU 空闲
  2. 只占用明确需要的 GPU：用 CUDA_VISIBLE_DEVICES 显式选择
  3. 训练结束后立即释放：不要让空闲进程占着显存
  4. 长时间训练（>24h）在群里通知：告知占用的 GPU 编号和预计时长
  5. 不抢占别人还在用的 GPU：即使利用率很低也先问一声
  6. 不要同时占用全部 GPU：给其他人留至少 1-2 块
  7. 训练完了在群里说一声：让排队的人可以接着用

4. ⚡ 进程与资源管理

4.1 tmux 使用规范（强制）

任何预计运行超过 10 分钟的任务，必须放到 tmux/screen 中运行。否则你的 SSH 一断，进程就没了。
# 命名规范：用你的名字+任务名，方便别人识别
tmux new -s zhangsan-train-resnet    # ✅ 好
tmux new -s test                     # ❌ 太模糊，不知道是谁的

# 完整工作流
tmux new -s myname-task
conda activate torch-gpu
CUDA_VISIBLE_DEVICES=3 python train.py
# Ctrl+B, D → 安全脱离（训练继续跑）

# 下次回来看：
tmux ls                    # 列出所有 session
tmux attach -t myname-task # 重新连接

# 训练完成 → 清理你的 session！
tmux kill-session -t myname-task     # 或用 exit

4.2 CPU 与内存管理

# 查看自己占用的资源
top -u $(whoami)

# 查看自己的所有进程
ps aux | grep $(whoami)

# 批量清理自己的僵尸进程
ps aux | grep $(whoami) | grep defunct | awk '{print $2}' | xargs kill

# DataLoader workers：不要超过 CPU 核心数/GPU 数量
n_cpu=$(nproc)
n_gpu=$(nvidia-smi -L | wc -l)
# 单 GPU 训练：num_workers 设为 4-8 即可
# 多 GPU 训练：num_workers 不超过 n_cpu/(2*n_gpu)

4.3 后台任务规范

# ✅ tmux（首选！）
tmux new -s task; python train.py; Ctrl+B D

# ✅ nohup（备用，简单的单次任务）
nohup python train.py > train.log 2>&1 &
# 退出后进程继续运行，输出重定向到 train.log

# ❌ 直接 &（ssh 断开后进程可能被 kill）
python train.py &

# 查看后台任务
jobs -l

• 代码只在服务器上存着，没 push 到 git → 服务器磁盘故障 → 代码永远丢失

6.2 误删恢复

# 如果文件刚被 rm 删除（且是 ext4 文件系统）
# ① 立刻停止对该分区的一切写操作！
# ② 尝试用 extundelete 或 testdisk 恢复
sudo extundelete --restore-file /home/user/deleted_file.txt /dev/sda1

# 但恢复成功率不是 100%。最好的保护是：
# ✅ 代码用 git 管理
# ✅ 重要数据定期备份
# ✅ 用 trash-cli 替代 rm（有回收站）

7. 🚀 效率提升技巧

7.1 推荐别名配置（~/.bashrc）

# === 安全 rm（移到回收站而非直接删除）===
alias rm='rm -i'                 # 删除前确认
# 或装 trash-cli: pip install trash-cli; alias rm='trash-put'

# === GPU 快速查看 ===
alias gs='gpustat -apu'          # ⭐ 最常用
alias gsw='watch -n 1 gpustat -cP'  # 实时监控

function gpu-watch() {
    ssh "$1" "watch -n 2 'echo === GPU ===; gpustat -cP; echo; echo === Top CPU ===; ps aux --sort=-%cpu | head -6'"
}

function gpu-usage() {
